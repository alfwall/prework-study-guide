HTML


  Class Material


  • "Semantic" HTML elements clearly describe its meaning to both browser and developer.
    • <header>, <main>, and <footer> are the most obvious semantic elements.
    • <section>, <aside>, <nav, <article>, <figure>, <figcaption>, and many others exist! Look em up sometime.
  • "Non-Semantic" HTML elements don't tell us anything about what could be within them.
    • <div> and <span> are the most common, alongside <font> and <table>.
  • The <head> element contains information about the webpage.
  • The <body> element represents the visible content shown to the user.
  • Within body are <header>, <main>, and <footer> child elements.
  • Elements are tags plus contents, whereas tags are just what's in the angle brackets.
  • Header tags range from <h1>, <h2>, to <h6>, decreasing in size.
    • <h1> should only be used once per page, for accessibility.
  • Lists can be used to spread out info, like in this segment.
    • <ol> and <ul> denote Ordered and Unordered Lists, respectively.
    • Lists of any sort contain <li> List Items.
    • Sub-lists can be displayed by adding a whole new <ol> or <ul> alongside list elements.

  Personal Research


  • Displaying special characters like < and > requires "HTML character entities".
    • Character Entities are required to not conflict with reserved characters when parsing markup.
    • They all start with Ampersand (&) and end with SemiColon (;)
    • Some include a Pound (#) sign to denote numbers. But it can also be used for symbols. Not reasonable to memorize.

CSS

  • Cascading Style Sheets are how webpages get styled (and don't look like piles of weirdly formatted text).
  • CSS can style elements "inline" or use internal/external style sheets.
    • Inline styling involves adding directly to individual elements.
      • <h1 style="color:red;">Dummy Text</h1>
    • Internal style sheets are stored within the HTML document itself, nested inside <style> elements stored within the <head>.
    • External style sheets are in a separate file. Referenced through <link> elements within <head>.
  • CSS looks like this: [selector] {property: value; property: value;}
    • [selector] can be tag type, tag ID, tag class, or more! Asterisk (*) means ALL ELEMENTS.
    • property/value pairs are individual "Declarations".
  • "Margin" refers to space OUTSIDE an element. A margin of 0 indicates NO extra space around an element.
  • "Padding" refers to space surrounding the Content within an element. More padding means bigger element.
  • Elements contain Content. Element size is determined by Content and Padding. Elements are spaced from one another based on Margin.
    • Look up "CSS Box Model" for visuals.

Git

  • "Version Control" is a method of maintaining a timeline of changes across a project.
  • Git is a relatively reasonable standard tool for version control, though it has quirks.
  • GitHub, on the other hand, is a website that hosts Git repo's on the cloud.
  • We connect to repos securely using SSH keys.
  • git config contains settings for individual users interacting with a repo.
    • FIRST TIME SETUP, YOU SHOULD USE THESE! It'll make clear whose commits are whose, and how you might contact someone about a commit they made.
    • git config user.name [name]: Prints name of current user, or updates it if [name] is provided.
    • git config user.email [email]: Prints email of current user, or updates it if [email] is provided.
  • git clone [git@github:repo_name.git]: Download a copy of the repo.
  • git status: checks what branch we are currently on, and if there are changes.
  • git pull origin [source]: Download the most recent updates from [source].
    • git pull origin MAIN: the most common use case.
  • git checkout -b branch-name: creates a new branch and switches to it
  • git add -A: Adds changes in the working directory to the staging area.
    • -A implies "All Changes", but I like to use . for "Everything Here"
  • git commit -m "A commit message!": Creates a snapshot of changes to the repo.
  • git push origin [branch]: Upload commits from local repo to remote repo.
    • "git push origin main" is considered bad practice, so push to feature branches instead. Merge afterwards.
